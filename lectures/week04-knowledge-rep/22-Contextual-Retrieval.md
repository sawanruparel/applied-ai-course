# Contextual Retrieval

Anthropic's approach (September 2024): use an LLM to add context to each chunk **before** embedding it.

### The Problem

A chunk like "The company was founded in 1983" is meaningless without knowing **which** company. Standard chunking strips away the surrounding context that makes a chunk interpretable.

### The Approach

For each chunk, prepend a short contextual description generated by an LLM:

**Before** (raw chunk):
> "Revenue increased 15% year-over-year, driven primarily by expansion in the APAC region."

**After** (contextualized chunk):
> "This chunk is from Acme Corp's Q3 2025 earnings report, specifically the revenue breakdown section. Revenue increased 15% year-over-year, driven primarily by expansion in the APAC region."

### Implementation

1. For each chunk, send the **full document + the chunk** to an LLM
2. Prompt: "Give a short context (1-2 sentences) situating this chunk within the overall document"
3. Prepend the generated context to the chunk before embedding
4. Use prompt caching to avoid re-processing the full document for each chunk (major cost savings)

### Results (Anthropic's Benchmarks)

| Method | Recall Improvement vs Baseline |
|--------|-------------------------------|
| Contextual embeddings only | 35% reduction in retrieval failures |
| Contextual embeddings + BM25 hybrid | 49% reduction in retrieval failures |
| Contextual embeddings + BM25 + reranking | 67% reduction in retrieval failures |

### Practical Considerations

- **Cost**: one LLM call per chunk at ingestion time (use prompt caching and a fast model)
- **Latency**: ingestion is slower; retrieval is unchanged
- **When to use**: high-value corpora where retrieval quality is critical
- **When to skip**: rapidly changing data where re-ingestion cost is prohibitive
- Works with any embedding model -- it improves the text being embedded, not the model itself
